Architecture

Social Login Implementation Architecture

Status: Draft v0.1 Last reviewed: 2026-07-02 Branch: docs/social-login-functional-spec Source: docs/social-login-functional-spec.html

Architecture Summary

Intervals should use server-owned authentication. The ASP.NET Core API owns provider redirects, provider callbacks, account creation, account linking, session cookies, and API authorization. The React app owns only the login screen, route gating, and authenticated UI state.

The first release supports Google and X. Apple is deferred to phase 2. There is no email/password signup and no email magic-link fallback. Practice progress remains local in this auth release. Authentication persistence uses PostgreSQL modeled and started by Aspire during local development.

Current System Context

  • Frontend: Vite and React in src/, currently loaded as one training app.
  • Backend: ASP.NET Core minimal API in api/Intervals.Api/Program.cs.
  • Orchestration: the repo currently uses file-based apphost.cs. The target auth architecture should move to a project-based AppHost so local orchestration and Aspire.Hosting.Testing use the same resource graph.
  • Current API surface: GET /api/status and /health.
  • Current persistence: none. Authentication requires adding an Aspire-managed PostgreSQL database for users and provider links.

Core Decisions

Concern Decision Reason
Authentication owner ASP.NET Core API Keeps OAuth codes, provider tokens, cookies, account linking, and authorization out of browser code.
Browser session HTTP-only secure cookie Best fit for same-origin Vite plus API deployment. No provider tokens are stored in browser storage.
Remember me Persistent cookie when selected Satisfies the selected product behavior while keeping a shorter default session for users who do not opt in.
Account data EF Core data model User and provider-link persistence is required for returning login and future account linking.
Initial database Aspire-managed PostgreSQL Aspire already orchestrates Dockerized local resources, so the implementation should use the same database engine locally that it expects to use beyond local development.
AppHost form Project-based AppHost Enables Aspire.Hosting.Testing to launch the real distributed app in tests while keeping the AppHost as the single source of local resource wiring.
Provider secrets .NET user secrets for this phase Matches the current decision. Production still needs managed secret storage before public launch.
Cloud progress sync Out of scope Authentication identifies users now. Server-side progress can be designed after auth is stable.

Component Architecture

React Web App

  • Renders /login and authenticated training UI.
  • Calls GET /api/session on startup to determine auth state.
  • Posts login intent to /auth/login/{provider} with returnUrl and rememberMe.
  • Shows user-safe messages for cancelled or failed provider login.
  • Never handles provider access tokens or ID tokens.

ASP.NET Core API

  • Configures cookie authentication and external providers.
  • Owns login start, callback, logout, and session endpoints.
  • Uses fallback authorization to deny anonymous API access.
  • Creates internal users and provider links.
  • Emits auth audit events without logging tokens.

Auth Database

  • Stores internal users.
  • Stores external provider links.
  • Stores minimal auth event audit records.
  • Does not store provider tokens in the initial release.

External Providers

  • Google: OAuth/OIDC identity provider.
  • X: OAuth 2.0 authorization code with PKCE.
  • Apple: phase 2 provider slot, not visible in MVP UI.
  • Scopes are limited to identity and email data.

Target AppHost Project

Replace the root file-based apphost.cs with a project-based AppHost before adding Aspire distributed-app tests. This keeps Aspire as the single orchestration source for local development and gives the test project a concrete AppHost project reference. 

apphost/
  Intervals.AppHost.csproj
  Program.cs
  • Intervals.AppHost.csproj uses Aspire.AppHost.Sdk.
  • Program.cs contains the current Vite/API wiring plus the new PostgreSQL resource model.
  • The AppHost project references or hosts the existing API project and keeps Aspire.Hosting.JavaScript for AddViteApp().
  • The AppHost project adds Aspire.Hosting.PostgreSQL for AddPostgres() and AddDatabase().
  • The Aspire test project references Intervals.AppHost.csproj, which enables DistributedApplicationTestingBuilder.CreateAsync<...> through the generated Projects namespace.

Aspire Resource Model

Local development should use Aspire's first-party PostgreSQL hosting integration instead of a file-based or in-memory database. Aspire should start the PostgreSQL container, manage host, port, password, and connection string, and inject the database connection into the API through WithReference(). 

var postgres = builder.AddPostgres("postgres")
    .WithDataVolume("intervals-postgres-data")
    .WithLifetime(ContainerLifetime.Persistent);

var intervalsDb = postgres.AddDatabase("intervalsdb");

var api = builder.AddProject("api", "api/Intervals.Api/Intervals.Api.csproj")
    .WithReference(intervalsDb)
    .WaitFor(intervalsDb)
    .WithHttpEndpoint()
    .WithHttpHealthCheck("/health")
    .WithExternalHttpEndpoints();
  • The API should consume the injected ConnectionStrings__intervalsdb value through the Aspire Npgsql EF Core integration: builder.AddNpgsqlDbContext<IntervalsDbContext>("intervalsdb").
  • Do not model PostgreSQL host, port, username, password, or connection string manually with WithEnvironment().
  • Keep a persistent data volume so local accounts and provider links survive AppHost restarts.
  • If a stale persistent PostgreSQL volume causes password authentication failures after AppHost recreation, remove the stale container and volume, then let Aspire recreate them.

Backend Middleware Pipeline

var builder = WebApplication.CreateBuilder(args);

builder.AddNpgsqlDbContext<IntervalsDbContext>("intervalsdb");
builder.Services.AddAuthentication(...);
builder.Services.AddAuthorization(options =>
{
    options.FallbackPolicy = new AuthorizationPolicyBuilder()
        .RequireAuthenticatedUser()
        .Build();
});
builder.Services.AddAntiforgery(...);
builder.Services.AddRateLimiter(...);
builder.Services.AddHealthChecks();

var app = builder.Build();

app.UseForwardedHeaders();
app.UseHttpsRedirection();
app.UseRateLimiter();
app.UseAuthentication();
app.UseAuthorization();

app.MapAuthEndpoints();
app.MapGet("/api/session", ...).RequireAuthorization();
app.MapGet("/api/status", ...).RequireAuthorization();
app.MapHealthChecks("/health").AllowAnonymous();

app.Run();

/health may be anonymous only when it is private to Aspire, a load balancer, or platform health checks. If the endpoint is publicly reachable, protect it at the platform layer or require authentication.

Authentication Configuration

Cookie Schemes

  • Main application cookie: Intervals.App, HTTP-only, secure in non-development, SameSite=Lax.
  • External temporary cookie: used only during provider callback flow.
  • Default session lifetime: configurable, recommended 8 hours.
  • Remember-me lifetime: configurable, recommended 30 days.
  • Sliding expiration: enabled for app cookie, bounded by the persistent remember-me expiration.

Provider Schemes

Scheme Handler Implementation Notes
Google Google or OIDC provider middleware Configure client id and secret from user secrets. Request openid, email, and profile. Use the stable provider subject claim as ProviderUserId.
X Generic OAuth handler with PKCE Configure authorization endpoint, token endpoint, userinfo endpoint, users.read, and users.email. Set UsePkce = true. Map the stable X user id as ProviderUserId.

Data Model

Tables

Table Fields Constraints
AppUsers Id, DisplayName, Email, EmailNormalized, AvatarUrl, CreatedUtc, LastLoginUtc, DisabledUtc Primary key on Id. Index on EmailNormalized for support and merge guidance, but do not make email globally unique in MVP.
ExternalLogins Id, UserId, Provider, ProviderUserId, Email, EmailVerified, DisplayName, AvatarUrl, CreatedUtc, LastLoginUtc Foreign key to AppUsers. Unique index on Provider plus ProviderUserId.
AuthEvents Id, UserId, Provider, EventType, OccurredUtc, Success, FailureCode, CorrelationId Optional UserId. No raw provider payloads, tokens, authorization codes, or secrets.

Entity Rules

  • Internal user ids are immutable and become the application identity.
  • Provider name plus provider user id is the login identity key.
  • Email is useful for display and account recovery support, but not trusted as the only account key.
  • Provider access tokens are not stored in the initial release.
  • If a provider login returns an email matching another user, do not merge automatically. Require an explicit account-linking flow later.

Backend Endpoints

Method Path Anonymous? Purpose
POST /auth/login/{provider} Yes Validates provider, return URL, and remember-me flag, then starts the external provider challenge.
GET /auth/callback/{provider} Yes Completes provider login, creates or updates account records, issues the app cookie, and redirects to the safe return URL.
POST /auth/logout No Invalidates the app cookie and redirects to login.
GET /api/session No Returns current authenticated user summary and enabled providers.
GET /api/status No Existing status endpoint, now requiring auth.
GET /health Private infrastructure only Health probe for Aspire or hosting infrastructure, not a public anonymous app endpoint.

Login Flow

  1. User visits /. React calls GET /api/session.
  2. API returns 401. React navigates to /login?returnUrl=/.
  3. User selects Google or X and optionally selects Remember me.
  4. React posts to /auth/login/{provider} with a local relative return URL and remember-me value.
  5. API validates input, stores protected auth properties, generates provider state, and redirects to the provider.
  6. Provider authenticates the user and redirects to /auth/callback/{provider}.
  7. API validates state, exchanges code, reads identity claims, and creates or updates AppUsers and ExternalLogins.
  8. API issues the application cookie with persistent expiration only when remember-me was selected.
  9. API redirects to the safe return URL. React reloads session state from /api/session.

Authorization Model

  • Configure a fallback authorization policy requiring authenticated users.
  • Apply AllowAnonymous() only to login start, provider callbacks, static login assets, and private infrastructure health.
  • Apply RequireAuthorization() explicitly to all /api/* endpoints for readability.
  • Reject anonymous API requests with 401. Use 403 only for future authenticated users who lack a required permission.
  • Include internal user id in the auth cookie claims as sub or a namespaced application claim. Do not use email as the primary authorization subject.

Frontend States

State UI Behavior

Anonymous Show login page only.
Authenticated Render training UI and account/logout controls.
Provider cancelled Return to login page with a neutral retry message.
Provider error Show a user-safe error and support correlation id when available.

Security Controls

  • Validate all return URLs on the backend. Accept only relative local paths such as / or /?mode=mixed.
  • Use provider state and PKCE where supported. X must use PKCE.
  • Do not store provider access tokens, refresh tokens, authorization codes, ID tokens, or raw provider payloads.
  • Protect future state-changing API calls with antiforgery controls because browser auth uses cookies.
  • Configure forwarded headers before auth in deployed environments so redirect URI generation uses the correct external scheme and host.
  • Rate-limit login start, callback failure, logout, and session endpoints.
  • Persist ASP.NET Core data-protection keys in production so cookies remain valid across restarts and multiple instances.
  • Log auth events with correlation ids, but never tokens, secrets, or provider authorization codes.

Aspire and Local Development

  • Add the first-party PostgreSQL hosting integration to the AppHost and model an intervalsdb database for auth data.
  • Use WithReference(intervalsDb) and WaitFor(intervalsDb) on the API resource so Aspire injects the connection string and starts the API only after PostgreSQL is ready.
  • Use a persistent PostgreSQL data volume for local development so user accounts and provider links survive AppHost restarts.
  • Keep provider credentials in .NET user secrets for api/Intervals.Api.
  • Register provider callback URLs for the API endpoint used during local development. Example: https://localhost:<api-port>/auth/callback/google and https://localhost:<api-port>/auth/callback/x.
  • Vite should call the API through the same origin/proxy path used by Aspire where possible, so cookies behave like production.
  • If Aspire exposes separate web and API origins locally, configure CORS only for the known Vite origin and allow credentials.
  • Before public production launch, replace user secrets with the hosting platform's managed secret store and register final public callback URLs.

Testing Strategy Decision

Use two complementary integration-test styles. For backend auth behavior, the default is WebApplicationFactory plus Testcontainers.PostgreSql. For full-system smoke and end-to-end tests, use Aspire testing through the project-based AppHost.

Approach Use For Reason
WebApplicationFactory + Testcontainers API auth integration tests: anonymous rejection, session creation, logout, account creation, provider-link persistence, and database constraints. Fast enough for routine test runs, gives direct control over API test configuration, and exercises real PostgreSQL semantics.
Aspire.Hosting.Testing Distributed-app tests that start the AppHost, API, web frontend, and PostgreSQL together. Verifies the real Aspire resource graph, service discovery, startup ordering, and cross-resource behavior.

Architecture decision: convert the current file-based apphost.cs to a project-based AppHost. DistributedApplicationTestingBuilder requires a project-based AppHost reference, and this conversion makes the same AppHost resource graph available to both local development and full-system tests.

Scope boundary: DistributedApplicationTestingBuilder requires a separate-process, closed-box style. It is excellent for proving the resource graph works, but it is not the best tool for detailed auth edge cases that need fake auth schemes, direct API configuration, or precise database reset control.

Implementation Risks

Risk Mitigation
X email availability may vary by app access and user account. Treat email as optional. Use X provider user id as the durable identity.
Open redirect through returnUrl. Accept only local relative paths and reject URLs with schemes, hosts, backslashes, or protocol-relative prefixes.
Auth cookie broken behind proxy. Configure forwarded headers and provider callback URLs for the real external origin.
Anonymous API accidentally added later. Use fallback authorization plus tests proving anonymous /api/* requests return 401.
User loses access to only linked provider. Accepted for MVP. Add provider linking or recovery in a later account-management phase.

Done Criteria

  1. Signed-out users can reach only login and auth flow endpoints.
  2. Signed-out /api/status and /api/session return 401.
  3. Google login creates and reuses an internal user account.
  4. X login creates and reuses an internal user account.
  5. Remember-me creates a persistent session; normal login creates the standard session.
  6. Logout invalidates API access until the user signs in again.
  7. Provider secrets are stored in .NET user secrets locally and never committed.
  8. Provider tokens, auth codes, and secrets do not appear in browser storage, API responses, or logs.
  9. Automated backend and frontend tests cover the protected auth boundary and core session behavior.
