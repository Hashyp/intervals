Intervals implementation improvement plan

This document captures a multi-pass review of the current frontend and backend auth implementation, with emphasis on duplication, security risks, and UI/accessibility improvements. Because there is no callable subagent-spawn tool in this session, the requested handoff was simulated as three independent passes: draft reviewer, validation reviewer, and implementation planner.

P1Protect OAuth login/link start POSTs with antiforgery validation.Prevents cross-site initiation of auth/link flows and aligns all auth mutations behind the same CSRF policy.

Implementation plan

1. Backend hardening foundation

1.1 Add shared helpers.
  • Create an IAuthEventRecorder service for AuthEvents.
  • Create one email normalization helper with max-length validation.
  • Create an endpoint filter/helper for antiforgery validation returning the existing ApiError shape.
1.2 Protect OAuth start endpoints.
  • Validate antiforgery in /auth/login/{provider} and /auth/providers/link/{provider}.
  • Load a request token in auth/account pages and include it in native form posts as the antiforgery form field.
  • Add integration tests proving missing tokens return 400 and valid tokens still challenge/redirect.
1.3 Enforce merge-cookie expiry.
  • Use ITimeLimitedDataProtector or include an expiresUtc value in the protected payload and validate it on read.
  • Add tests for valid, expired, tampered, and primary-mismatch pending-merge cookies.
1.4 Partition rate limits.
  • Replace AddFixedWindowLimiter global policies with AddPolicy and RateLimitPartition.GetFixedWindowLimiter.
  • Partition generic auth by authenticated user ID when present, otherwise remote IP; partition password reset by remote IP plus normalized email where practical.
  • Keep response behavior at 429 and add targeted tests for independent partitions.
1.5 Configure forwarded headers safely.
  • Do not clear known proxies/networks unconditionally in production.
  • Add configuration for trusted proxy IPs/networks; allow trust-all only in development/testing or behind an explicit opt-in.

2. Frontend consolidation and UI polish

2.1 Consolidate remember-me.
  • Define a social-provider config array and render provider forms from it.
  • Render exactly one Remember me checkbox near the login actions.
  • Write the shared value into each social form via hidden rememberMe input and into password login JSON.
  • Update tests to assert one visible checkbox, all providers include the current value, and X supports remembering too.
2.2 Add antiforgery token support for native forms.
  • Create useAntiforgeryToken or preload through AuthProvider.
  • Disable native provider/link buttons while the token is loading, or show a retryable error if token loading fails.
  • Add hidden __RequestVerificationToken fields to social login/link forms.
2.3 Extract UI primitives.
  • Add AuthCard, AuthMessage, AuthField, PasswordConfirmFields, and optional SubmitButton components.
  • Refactor login, forgot/reset password, merge confirm, and account settings incrementally to avoid a large risky rewrite.
2.4 Complete CSS and accessibility polish.
  • Add styles for .auth-providers, .auth-providers__item, .auth-form--inline, .auth-form-actions, .auth-merge-accounts, and .account-bar__verify.
  • Move inline styles out of EmailVerificationBanner.
  • Add :focus-visible styles for auth buttons, links, and account-bar actions.
  • Replace window.confirm unlink flow with an accessible in-page confirmation pattern when time permits.
2.5 Reduce token exposure.
  • In ResetPasswordPage, capture the token once and immediately replaceState to remove it from the address bar.
  • Keep the token only in component state for the reset submission.
  • Add tests proving the token is submitted while the visible URL no longer contains it.

Suggested test checklist

  • npm test after frontend refactors.
  • dotnet test after backend helper/rate-limit/antiforgery changes.
  • Manual browser pass: login page visual layout, one remember-me control, keyboard focus order, provider form posts, account-settings provider list, merge confirmation.
  • Security regression pass: missing CSRF on all POST/DELETE auth endpoints, expired pending-merge cookie, partitioned rate limits, reset-token URL scrubbing.
