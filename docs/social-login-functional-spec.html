Functional specification

Social Login and Auth Gate for Intervals

Status: Draft v0.1 Last reviewed: 2026-07-02 Branch: docs/social-login-functional-spec

Executive Recommendation

Launch with Google and X (x.com) because they are explicit product requirements. Defer Apple to phase 2. Treat Facebook, Microsoft, and GitHub as later optional additions unless user research shows strong demand.

The app must not allow anonymous product access. Only the login page, auth start endpoints, auth callbacks, logout completion page, static assets required by login, and explicitly private infrastructure health checks may be reachable before authentication.

Goals

  • Require a signed-in account before a user can access the Intervals training experience.
  • Provide social login through Google and X at minimum, with a clear path for Apple and optional future providers.
  • Secure the API server-side so every public API endpoint rejects anonymous requests.
  • Create or reuse one internal Intervals user account per person, regardless of which linked provider they use later.
  • Avoid custom OAuth logic in the browser. Provider tokens must be exchanged and stored only by the backend when storage is required.

Non-Goals

  • Building paid subscriptions, roles beyond basic authenticated user, or organization/team accounts.
  • Requesting provider permissions to post, read timelines, manage contacts, or access unrelated social data.
  • Implementing email/password signup in the initial social-login release.
  • Migrating local anonymous practice history into a cloud account, except through a later explicit data migration feature.
  • Cloud-synced progress, cross-device practice history, and server-side score storage in the initial authentication release.

User Experience

Unauthenticated Visit

  • Visiting the root URL or any training route while signed out redirects to /login.
  • The app stores the originally requested path as a validated relative returnUrl.
  • No training UI, score data, or API-backed content is displayed before authentication.

Login Page

  • Show the Intervals name, a concise sign-in prompt, and provider buttons for Google and X.
  • Keep the layout ready for Apple in phase 2 without showing an inactive Apple button in the MVP.
  • Provider buttons must follow each provider's branding rules.
  • Include links to privacy policy, terms, and support contact.

Successful Login

  • First-time users get an internal Intervals account automatically.
  • Returning users resume the requested route or default to the training screen.
  • The header or account menu shows the display name and a logout action.

Failed Login

  • Cancelled provider login returns to /login with a neutral message.
  • Provider errors show a retry option and do not expose raw provider exception details.
  • If an email is already attached to another account, the user is guided through account linking after authenticating that account.

Functional Requirements

Authentication Boundary

  1. The training experience is available only to authenticated users.
  2. The API rejects anonymous requests with 401 Unauthorized for every public endpoint, including /api/status.
  3. The API returns 403 Forbidden when an authenticated user lacks permission for a future protected action.
  4. Browser route guards are not the security boundary. Authorization is enforced by the backend.
  5. Static assets required for rendering the login page may be public. The authenticated app shell must either be protected by the backend or contain no usable product data until session validation succeeds.

Session

  1. The backend issues a secure, HTTP-only session cookie after provider callback validation succeeds.
  2. Cookies use Secure, HttpOnly, and an appropriate SameSite value that supports OAuth callback redirects.
  3. Sessions expire after a configurable idle timeout and absolute lifetime.
  4. The login page offers a Remember me option. When selected, the backend issues a longer-lived persistent session for that browser. When not selected, the backend uses the standard session lifetime.
  5. Logout invalidates the server session and returns the user to /login.
  6. The React app reads session state from GET /api/session instead of decoding provider tokens in the browser.

Account Model

  1. Each account has an internal immutable user id.
  2. Each linked provider stores provider name, provider user id, display name, email when available, email verification status when available, avatar URL when available, first linked timestamp, and last login timestamp.
  3. Provider user id plus provider name is the unique login key.
  4. Email is used for display and merge guidance, not as the only identity key.
  5. Users can link another provider from account settings after signing in.
  6. Users cannot unlink their only active login provider unless another login method has already been linked.

Provider Flow

  1. POST /auth/login/{provider} starts login for a known enabled provider.
  2. The backend generates and validates state for CSRF protection.
  3. Flows that support PKCE use PKCE.
  4. The backend validates callback parameters, exchanges the authorization code, fetches the minimum user profile, and creates or updates the provider link.
  5. Requested scopes are limited to identity and email. Social posting, timelines, contacts, and write scopes are disallowed.

API Contract

  1. GET /api/session returns the current user summary for authenticated users.
  2. GET /api/session returns 401 for anonymous users.
  3. Existing endpoints, starting with /api/status, require authorization.
  4. API errors use a consistent JSON shape with code, message, and correlation id.
  5. Cross-origin requests with credentials are allowed only from approved app origins.

Security and Privacy Requirements

  • Use maintained authentication libraries or platform middleware for OAuth and OIDC.
  • Store client secrets only in local user secrets, environment variables, or a production secret store. Never commit secrets to the repository.
  • Do not store provider access tokens unless a concrete product feature requires calling provider APIs after login.
  • If provider tokens are stored later, encrypt them at rest and record purpose, scope, expiration, and revocation behavior.
  • Protect state-changing API calls with CSRF defenses when cookie authentication is used.
  • Log auth events such as login success, login failure, logout, provider link, and provider unlink without logging raw tokens or secrets.
  • Include rate limiting for login start, callback failure, and session endpoints.
  • Collect only profile data required for account display and support.

Current Repo Fit

The current project has a Vite/React frontend and a .NET minimal API. The recommended implementation posture is server-owned authentication: the API handles provider redirects and session cookies, while React treats /api/session as the source of truth.

  • In ASP.NET Core, configure a fallback authorization policy so new endpoints require authentication by default.
  • Mark only auth endpoints and private infrastructure endpoints with explicit allow-anonymous behavior.
  • Apply RequireAuthorization() to existing minimal API routes during migration.
  • Provider credentials should be read from .NET user secrets during development and initial validation. The Aspire AppHost must not hardcode them. Before public production launch, move the same values to the deployment platform's managed secret store.

Health Check Decision

The user requirement says no anonymous API access. Production health checks sometimes need unauthenticated probing by a load balancer. If this app needs that, the health endpoint must be private to the platform network or protected by platform-level authentication. It should not be a public anonymous endpoint.

Acceptance Criteria

  1. A signed-out visitor to / sees the login page and cannot access the training UI.
  2. Login buttons exist for Google and X, and each starts the correct provider flow.
  3. A successful first-time provider login creates one internal user account and signs the user in.
  4. A successful returning provider login signs in to the same internal account.
  5. A signed-out request to /api/status returns 401.
  6. A signed-in request to /api/status succeeds.
  7. Logout clears the session and prevents subsequent API access until the user signs in again.
  8. Provider access tokens and secrets never appear in browser storage, logs, or committed files.
  9. Login failure, provider cancellation, and provider outage states show user-safe messages on the login page.
  10. Automated tests cover anonymous rejection, authenticated API access, login callback success, callback state mismatch, and logout.

Testing Decision

  • API integration tests should use WebApplicationFactory with test authentication schemes and Testcontainers.PostgreSql for a disposable PostgreSQL database. This is the default approach for testing auth persistence, cookie behavior, logout, and anonymous API rejection against the same database engine used locally.
  • Aspire's Aspire.Hosting.Testing package is appropriate for full distributed-app tests that launch the AppHost, API, web frontend, and PostgreSQL together.
  • Decision: convert the current file-based apphost.cs to a project-based AppHost as part of the auth implementation plan so Aspire distributed-app tests can reference it. This adds full-system coverage, but it does not replace the more focused Testcontainers.PostgreSql API integration tests.

Resolved Product Decisions

  1. Apple is not included in the first authenticated release. It is deferred to phase 2.
  2. No email magic-link fallback is required for the initial release. Users must sign in with an enabled social provider.
  3. Practice progress stays local for this authentication release. Login secures access and identifies the user, but cross-device progress sync, server-side score storage, and account-based practice history are separate future features.
  4. The login page includes a Remember me option. Users who select it get a longer-lived persistent session; users who do not select it get the standard session lifetime.
  5. Secret setup for this phase: use .NET user secrets for Google and X client ids and client secrets during development and initial validation. Register localhost backend callback URLs for development. Before a public production launch, move those same values to the deployment platform's managed secret store and register provider callback URLs under the final production origin, for example /auth/callback/google and /auth/callback/x.
