Feature spec and implementation plan

Microsoft Social Login and Email Login Extension

Status: Draft v0.1 Created: 2026-07-07 Repository: Intervals Scope: auth extension planning

Executive Summary

The next auth increment should add Microsoft as the third social login provider and then complete the missing email and password account lifecycle: email verification, password reset, account linking, and safe account merge handling.

Microsoft should follow the current Google and X pattern: the API owns provider redirects and callbacks, the browser never stores provider tokens, provider identity is persisted through ExternalLogin, and /api/session exposes a provider status row for the UI. Email and password extensions should build on the existing PasswordCredential model, with Mailpit added to the Aspire AppHost for local email capture.

Current Auth Baseline

Area Current implementation Planning implication
Providers Google and X are registered in api/Intervals.Api/Auth/AuthExtensions.cs. Password login is implemented separately through PasswordAccountService. Microsoft should be added as a peer provider, not as a special case in endpoint routing.
Provider names AuthProviderNames normalizes provider ids and maps them to authentication schemes. Add microsoft and Microsoft here, then let the existing /auth/login/{provider} and /auth/complete/{provider} flow handle it.
Sessions The API issues an HTTP-only app cookie and uses an external temporary cookie while provider authentication completes. Microsoft must use the same ExternalCookieScheme, SaveTokens = false, remember-me propagation, and remote-failure redirect behavior.
Persistence AppUser, ExternalLogin, PasswordCredential, and AuthEvent are modeled through EF Core and PostgreSQL. Microsoft does not require a schema change. Email verification, password reset, and account linking do.
Email/password Registration creates PasswordCredential.EmailVerified = false; login works before verification; reset and email confirmation flows do not exist yet. Verification can be introduced without breaking existing users, but product behavior must decide whether unverified users can continue signing in.
Account merge Tests assert that matching email alone does not auto-merge accounts. Preserve this rule. Linking and merge must be explicit and require proof of control of both accounts.
Local infrastructure The AppHost starts PostgreSQL and the Vite frontend. There is no local email capture resource yet. Add Mailpit through the C# AppHost in run mode and reference it from the API for local email verification and reset flows.

Goals and Non-goals

Goals

  • Add Microsoft sign-in for personal, work, and school Microsoft accounts.
  • Keep Microsoft login consistent with Google and X in routing, cookies, audit events, profile mapping, and frontend status.
  • Provide email verification for password accounts with local delivery through Mailpit.
  • Provide password reset with one-time, expiring, hashed reset tokens.
  • Provide account linking and safe merge paths without email-only auto-merge.
  • Identify adjacent missing auth subfeatures and sequence them by risk and value.

Non-goals

  • Do not store Microsoft, Google, or X access tokens after login.
  • Do not grant Microsoft Graph data access beyond identity and email information needed for sign-in.
  • Do not merge accounts only because two providers return the same email.
  • Do not replace the custom auth model with ASP.NET Core Identity in this increment.
  • Do not add production email provider credentials to source control or local appsettings.

User Experience

  • The login page shows Continue with Microsoft alongside Google and X.
  • The Microsoft form posts to /auth/login/microsoft with the same sanitized returnUrl behavior as the other providers.
  • Provider cancellation and provider errors use the existing auth=cancelled and auth=provider_error messaging.
  • After success, the user lands on the captured local return URL and /api/session includes Microsoft as linked.
  • Provider buttons should be visually distinct but use the same layout density, disabled states, and tests as the existing login page.

Backend Behavior

  • Add Microsoft.AspNetCore.Authentication.MicrosoftAccount to the API project, version-aligned with the current ASP.NET Core packages.
  • Register the provider only when Authentication:Microsoft:ClientId is present, matching the current optional provider behavior.
  • Use ExternalCookieScheme, CallbackPath = "/auth/callback/microsoft", and SaveTokens = false.
  • Use the same OAuthEvents.OnRemoteFailure handling as Google and X.
  • Extend ExternalProfileBuilderTests with Microsoft claims to lock down provider id, display name, email, verified state, and durable subject extraction.
  • Extend AuthIntegrationTests so /api/session reports Microsoft provider status.
Email claim caution. Microsoft identity tokens do not guarantee that an email claim is always present. Treat Microsoft email as optional. Do not use email alone as an account key or merge signal.

Email Login Extension Feature Spec

Email Verification

  • Registration sends a verification email to the password credential email address.
  • The verification token is one-time, random, expiring, and stored only as a hash.
  • Verification marks PasswordCredential.EmailVerified = true and sets a verified timestamp.
  • Users can request a resend with rate limiting and without leaking whether an address exists.
  • Unverified signed-in users see a compact banner with resend action.

Password Reset

  • Forgot-password accepts an email address and always returns a generic success response.
  • If the email has a password credential, send a reset link through the email sender.
  • The reset token is one-time, hashed at rest, and short-lived.
  • Resetting the password clears lockout state, invalidates outstanding reset tokens, and rotates a user security stamp.
  • Users can sign in with the new password immediately after reset.

Account Linking

  • Authenticated users can link Microsoft, Google, X, and password credentials from account settings.
  • Linking a new social provider uses the provider challenge flow with a protected mode=link property.
  • Adding a password to a social-only account requires email verification before treating that email as verified.
  • Unlinking is allowed only when the account has at least one remaining usable login method.

Safe Merge Path

  • If a provider or password credential already belongs to another user, do not silently attach it.
  • Require proof of control of the currently signed-in account and the target account before merge.
  • Show an explicit confirmation screen with both account identities and the resulting primary account.
  • Merge inside one transaction, move auth credentials to the primary user, preserve audit history, and mark the secondary account as merged instead of deleting it.

Recommended Product Decisions

Question Recommendation Reason
Can unverified email users sign in? Yes initially, with a persistent verification banner. This avoids breaking existing password accounts created before verification exists. Later, sensitive account actions can require verified email.
Can password reset be used before email verification? Yes, if the user controls the inbox at reset time. A successful reset link proves current inbox control and should also mark the password email as verified.
Should social provider email verify the account email? Only for that provider login record unless the provider supplies a clear verified email signal. Provider emails can be absent, aliases, or not equivalent to a local password credential email.
Should matching emails auto-link accounts? No. Existing tests already encode this security decision. Explicit linking is safer and auditable.
Should Mailpit be available in distributed tests? No by default; add it in AppHost run mode and use a fake sender in unit/integration tests. The current AppHost keeps dev-only UI resources out of smoke tests. Mailpit should follow that pattern unless an end-to-end email smoke test is explicitly added.

Data Model Plan

No Schema Change for Microsoft Login

Microsoft fits the existing ExternalLogins table: Provider = "microsoft", a durable ProviderUserId, optional email, optional display name, optional avatar URL, and last-login timestamps.

New Email and Account Action Token Table

Add a purpose-scoped token table instead of storing raw verification or reset tokens on the user row. 

AuthActionToken
  Id uuid primary key
  UserId uuid not null
  Purpose varchar(64) not null
  TokenHash varchar(128) not null
  Email varchar(320) null
  EmailNormalized varchar(320) null
  CreatedUtc timestamptz not null
  ExpiresUtc timestamptz not null
  ConsumedUtc timestamptz null
  RevokedUtc timestamptz null
  CorrelationId varchar(64) null
  • Generate at least 32 random bytes for the public token and encode with base64url.
  • Store only a SHA-256 or stronger hash of the token.
  • Use purpose values such as email_verification, password_reset, and account_merge.
  • Index Purpose plus TokenHash.
  • Invalidate older active tokens for the same user, purpose, and email when a new token is issued.

Recommended User and Credential Additions

  • Add PasswordCredential.EmailVerifiedAtUtc for auditing.
  • Add AppUser.SecurityStamp or SessionVersion to support session invalidation after password reset and future security events.
  • Add AppUser.MergedIntoUserId and AppUser.MergedUtc if safe merge ships in the same increment.
  • Keep PasswordCredentials.EmailNormalized unique, and keep AppUsers.EmailNormalized non-unique.

Endpoint Plan

Endpoint Auth Purpose Security notes
POST /auth/login/microsoft Anonymous Start Microsoft login. Existing provider login endpoint after provider map update.
GET /auth/complete/microsoft Anonymous Complete Microsoft login and issue app cookie. Uses external cookie and validated return URL.
POST /auth/email-verification/request Authenticated Send or resend verification email for the signed-in password account. CSRF protected, rate limited, generic success response.
GET /auth/email-verification/confirm Anonymous Consume verification token and mark email verified. One-time token, redirects to the app with a safe status code.
POST /auth/password/forgot Anonymous Request password reset email. No user enumeration; rate limited by IP and email hash.
POST /auth/password/reset Anonymous Consume reset token and set a new password. CSRF not required if token is bearer proof, but rate limit and validate password policy.
POST /auth/password/change Authenticated Change password from account settings. Require current password when a password credential already exists.
POST /auth/providers/link/{provider} Authenticated Start provider linking challenge. Stores link mode and current user id in protected auth properties.
GET /auth/providers/complete/{provider} Anonymous Complete provider linking or route to merge confirmation. Must not attach credentials that already belong to another user without explicit merge proof.
DELETE /auth/providers/{provider} Authenticated Unlink a provider from the current account. Reject when it would leave the account without any usable login method.

Mailpit Development Resource

Add Mailpit to the C# AppHost as a dev-only resource, consistent with the current pgAdmin and Vite run-mode pattern. The API should read Aspire-injected Mailpit environment variables and send email through a small IEmailSender abstraction. 

if (builder.ExecutionContext.IsRunMode)
{
    var mailpit = builder.AddMailPit("mailpit")
        .WithDataVolume("intervals-mailpit-data");

    api.WithReference(mailpit);
}
  • Add CommunityToolkit.Aspire.Hosting.MailPit to apphost/Intervals.AppHost.csproj, preferably through aspire add mailpit.
  • Keep Mailpit out of distributed smoke tests unless a specific email smoke test needs it.
  • Use MAILPIT_HOST and MAILPIT_PORT in development email sender configuration.
  • Expose Mailpit only through the Aspire dashboard in local development.
  • Use a fake in-memory sender in API tests for deterministic assertions.

Safe Account Linking Flow

  1. The signed-in user opens account settings and chooses a provider or password method to add.
  2. The API validates CSRF, verifies the current session, records mode=link, linkUserId, and a recent-auth timestamp in protected auth properties, then starts the provider challenge.
  3. On provider completion, the API builds an ExternalUserProfile and checks whether that provider id is already linked.
  4. If the provider is already linked to the same user, the operation is a no-op success.
  5. If the provider is unlinked, attach it to the current user and write an account_link_success event.
  6. If the provider belongs to another user, require explicit merge confirmation. The current session proves control of the primary account; the just-completed provider login proves control of the secondary credential.
  7. Merge in one transaction: lock both user rows, move external logins and password credential to the primary user if constraints allow, preserve auth events, mark the secondary user as merged, and emit an audit event.
  8. Redirect to account settings and refresh /api/session so the provider list reflects the new link state.

Implementation Plan

Phase 1: Microsoft Social Login

  1. Add the Microsoft authentication package to api/Intervals.Api/Intervals.Api.csproj.
  2. Add Microsoft constants and scheme mapping to AuthProviderNames.
  3. Register AddMicrosoftAccount in AddIntervalsAuth using the same conditional config pattern as Google and X.
  4. Set callback path, external sign-in scheme, token persistence policy, scopes, and remote failure behavior.
  5. Extend ExternalProfileBuilder and tests only if Microsoft emits claims not covered by the generic fallback.
  6. Add Microsoft to the /api/session provider status list.
  7. Update src/auth/sessionApi.ts provider type to include microsoft.
  8. Add the Microsoft login form and styling to LoginPage.tsx and styles.css.
  9. Extend frontend and backend tests for provider rendering, provider session status, unknown provider behavior, and callback failure behavior.
  10. Document Microsoft user secrets and redirect URI setup.

Phase 2: Email Delivery Foundation

  1. Add Mailpit hosting integration to the AppHost and wire it to the API in run mode.
  2. Add IEmailSender with development SMTP implementation and test fake implementation.
  3. Add email options for from address, app base URL, and token lifetimes.
  4. Add simple HTML and plain-text email templates for verification and reset messages.
  5. Add tests for email sender selection and template output without depending on external SMTP.

Phase 3: Email Verification

  1. Add AuthActionToken entity, EF configuration, and migration.
  2. Issue verification token during registration and on resend request.
  3. Add verification request and confirm endpoints.
  4. Mark password email verified on valid token consumption.
  5. Add session response fields or UI status handling needed for a verification banner.
  6. Add integration tests for token expiry, replay, resend, verified session output, and generic responses.

Phase 4: Password Reset

  1. Add forgot-password and reset-password endpoints.
  2. Use the same token table with password_reset purpose.
  3. Apply existing password policy during reset.
  4. Clear lockout and failed attempts after successful reset.
  5. Rotate security stamp or session version and invalidate outstanding reset tokens.
  6. Add login-page forgot-password and reset-password UI states.
  7. Add tests for generic forgot response, successful reset, expired token, replayed token, weak password, and login with the new password.

Phase 5: Account Settings, Linking, and Safe Merge

  1. Add account settings UI showing linked Google, X, Microsoft, and email/password methods.
  2. Add authenticated link-provider endpoints and provider completion routing for link mode.
  3. Add add-password and change-password flows for social-only and password users.
  4. Add unlink provider endpoint with last-login-method guard.
  5. Add merge confirmation and transaction service for provider collision handling.
  6. Add tests for no-op link, new link, collision requiring merge, confirmed merge, rejected merge, and unlink guard.

Phase 6: Hardening and Rollout

  1. Enforce DisabledUtc during social and password login.
  2. Record missing audit events such as login start, email verification sent, password reset requested, password reset completed, account linked, account unlinked, and account merged.
  3. Add rate limit buckets for resend verification and forgot-password separate from the general auth window.
  4. Review cookie invalidation behavior after password reset and account merge.
  5. Add production email provider abstraction without changing endpoint behavior.
  6. Run API tests, frontend tests, and AppHost smoke tests before enabling Microsoft in shared development.

Acceptance Criteria

  • Microsoft sign-in can create a new user and provider link.
  • Returning Microsoft sign-in reuses the existing provider link and user.
  • Google, X, Microsoft, and Email provider statuses are returned by /api/session.
  • Provider email collisions do not auto-merge users.
  • Email verification sends through Mailpit in local AppHost runs.
  • Email verification tokens cannot be replayed after consumption.
  • Password reset responses do not reveal whether an email exists.
  • Password reset tokens expire and cannot be replayed.
  • Account linking never steals a credential already attached to another user without explicit merge confirmation.
  • Unlinking cannot leave an account with no usable sign-in method.

Other Missing Auth Subfeatures

Priority Subfeature Why it matters Recommended timing
P1 Resend verification Required support path for users who miss or lose the first email. Ship with email verification.
P1 Change password Users who know their current password need a safer path than reset. Ship with password reset or account settings.
P1 Add password to social-only account Provides continuity if a social provider is unavailable or the user loses provider access. Ship with account linking.
P1 Disabled account enforcement AppUser.DisabledUtc exists but login currently needs explicit checks. Before public launch.
P1 Security-stamp/session invalidation Password reset and merge should invalidate stale sessions where appropriate. Before password reset ships.
P2 Change email Needed for password accounts when inbox ownership changes. After email verification and reset are stable.
P2 Provider unlink Users need control over linked providers, with last-method guard. With account settings.
P2 Security event notifications Notify users after password reset, email change, provider link, unlink, or merge. After email sender foundation exists.
P2 Auth event admin/support view Useful for debugging login failures and account support without exposing secrets. After audit event coverage improves.
P2 Production email provider Mailpit is local only; production needs a managed provider and secret storage. Before hosted beta.
P3 Passkeys or MFA Improves account security once basic recovery and linking flows are reliable. After account settings foundation.
P3 Device/session management Lets users revoke old sessions and inspect active sign-ins. After security-stamp support.
P3 Abuse controls CAPTCHA, IP reputation, or progressive throttling may be needed if auth endpoints are exposed publicly. After traffic patterns justify it.

Test Plan

Backend Tests

  • ExternalProfileBuilderTests covers Microsoft claims.
  • AuthIntegrationTests covers Microsoft provider status and provider collision behavior.
  • Password tests cover verification, reset, lockout reset, and generic forgot-password responses.
  • Account linking tests cover link, collision, merge, unlink, and last-method guard.
  • Token tests cover hashing, expiry, revocation, and replay prevention.

Frontend Tests

  • LoginPage.test.tsx checks Microsoft form action and button rendering.
  • Login page tests cover forgot-password and reset-password UI states.
  • Auth provider tests cover new provider id typing.
  • Account settings tests cover linked provider display and guarded unlink behavior.

Full end-to-end email tests can be deferred until the Mailpit resource is stable. Unit and API integration tests should use a fake email sender first so auth behavior remains deterministic.
